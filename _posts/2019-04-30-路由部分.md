> 路由介绍

这篇文章里，将从表象到原理讲解一下路由(Routing)。

## 什么是路由
静态网页的路由一般对应着文件的目录结构，就仿佛说如果访问网站test/1，那么网站下肯定有一个test的文件和test目录下的1.html文件。但是有了路由就不一样了，我们可以指定任意url到自己程序的某个位置。对应rails中就是任意url对应到任意controller的action。

而路由系统总共也就做这几件事：

1. 辨识HTTP Request的URL网址，然后对应到设定的Controller Action。

2. 处理网址内的参数字串，例如：/users/show/123送到Users controller的show action时，会将params[:id] 设定为 123

3. 辨识link_to和redirect_to的参数产生URL字串

在rails中路由能够识别url地址，并把他们分派给控制器动作或rack应用进行处理。

## 路由规则
在rails中路由文件为`config/routes.rb`。例如收到请求`GET /patients/17`的时候，会查询routes.rb中的路由，例如找到一条：
```ruby
get '/patients/:id', to: 'patients#show'
```
rails就会将该请求分配给patients控制器的show动作，同时把{id: '17‘}传入params。
当然Rails路由还可以生成路径和URL地址，如果将上面的代码改成：
```ruby
get '/patients/:id', to: 'patients#show', as: 'patient'
```
在控制器中有相应的代码：`@patient = Patient.find(20)`
对应视图中包含：`<%= link_to 'Patient Record', patient_path(@patient) %>`
呢么路由会生成路径/patients/20。

从上述中我们可以得知一般路径的写法是如下这样的：
```ruby
get 'meetings/:id', :to => 'events#show'
post 'meetings', :to => 'events#create'
```
如果有多条符合的规则，会优先使用最上面的规则。拿我们系列第一篇中的例子来看，他的路由形式是这样的(查看路由可以通过`rake routes`或者运行了`rails s`通过http://localhost:3000/rails/info/routes)进行访问：
![image.png](https://upload-images.jianshu.io/upload_images/7154520-a7f1d39195435854.png?imageMogr2/auto-orient/strip%7CimageView2/2/w/1240)
在这里我们可以看出由于只写了hello和goodbye方法，所以只有两条路由形式。

### 资源路由
资源路由允许我们为资源式控制器快速声明**所有**常见路由，这里的所有路由是指一些常见的动作：`GET、POST、PATCH、PUT 和 DELETE`。当rails应用收到`DELETE /photo/17`的时候会查询路由并把请求映射到控制器动作上，在此时我们发现routes.rb中存在资源路由：`resources :photos`，rails就会把请求分派给photos控制器中的destroy动作上，并把{id： ‘17’}传入params。
而在这里`resources :photos`相当于在rails应用中创建了7个不同的路由，这些路由都会映射到Photos控制器上：
![image.png](https://upload-images.jianshu.io/upload_images/7154520-ccbdf4965b0e2c0b.png?imageMogr2/auto-orient/strip%7CimageView2/2/w/1240)

同样拿系列中的第一篇文章举个例子，创建一个photos的controller，同时在routes.rb中加入`resources :photos`，然后查看路由：
![image.png](https://upload-images.jianshu.io/upload_images/7154520-5461a9a7d7470a8d.png?imageMogr2/auto-orient/strip%7CimageView2/2/w/1240)
如果利用单数路由就相当于默认动作少了一个index，所有的url helper也都为单数形式，显示出来的网址也是单数：`resource :photo`
如果不想启用这么多默认的7个resource路由，则可以通过expect或only参数：
```ruby
resources :events, :except => [:index, :show]
resources :events, :only => :create
```

我们甚至可以套叠路由：
```ruby
resources :projects do
  resources :tasks
end
```
这样的路径形式如下：`project_tasks_path(@project)`和`project_task_path(@proect, @task)`
它的链接地址如：`projects/123/tasks/123`;但是一般不建议把资源路由设计超过两层，因为这样会产生一些不必要的依赖关系。所以这时我们一般用浅嵌套路由：
```ruby
resources :products, shallow: true do
  resources :reviews
end
```

### Scope规则
scope方法可以让我们DRY我们的路由规则，将共通的controller、constraints、网址前置path和URL Helper前置名称移到scope成为参数。例如
```ruby
get 'foo/meetings/:id', :to => 'events#show'
post 'foo/meetings', :to => 'events#create'
```
可以改写成
```ruby
scope :controller => "events", :path => "/foo", :as => "bar" do
  get 'meetings/:id' => :show, :as => "meeting"
  post 'meetings' => :create	, :as => "meetings"
end
```
其中as会产生URL helper是bar_meeting_url和bar_meetings_url。

Module参数则可以让Controller分Module，例如
```ruby
scope :path => '/api/v1/', :module => "api_v1", :as => 'v1' do
  resources :projects
end
```
如此controller会是ApiV1::ProjectsController，网址如/api/v1/projects，而URL Helper如v1_projects_path这样的形式。

Namespace是Scope的一种特定应用，特别适合例如后台接口，这样就整组controller、网址path、URL Helper前置名称`都影响到：
```ruby
namespace :admin do
  resources :projects
end
```
如此controller会是Admin::ProjectsController，网址如/admin/projects，而URL Helper如admin_projects_path这样的形式。

在Namespace下也可以设定它的首页，例如：
```ruby
namespace :admin do
	root "projects#index"
end
```
http://localhost:3000/admin/ 就会使用ProjectsController index action了。
我们routes.rb中改写成这样，看看结果：
```ruby
Rails.application.routes.draw do
  namespace :admin do
    resources :photos
  end
  get 'say/hello'
  get 'say/goodbye'
end
```
![image.png](https://upload-images.jianshu.io/upload_images/7154520-31de95e5069616a9.png?imageMogr2/auto-orient/strip%7CimageView2/2/w/1240)
当然如果想不带前面的`/admin`前缀，可以如下声明：
```ruby
scope module: 'admin' do
  resources :articles, :comments
end
```

### 其他规则 
**特殊条件限定**

用:constraints设定一些参数限制，例如限制:id必须是整数。
```ruby
match "/events/show/:id" => "events#show", :constraints => {:id => /\d/}
```
另外也可以限定subdomain子网域：
```ruby
namespace :admin do
  constraints subdomain: 'admin' do
   		 resources :photos
  end
end
```
甚至可以限定IP位置：
```ruby
constraints(:ip => /(^127.0.0.1$)|(^192.168.[0-9]{1,3}.[0-9]{1,3}$)/) do
    match "/events/show/:id" => "events#show"
end
```

**自定群集路由Collection**
除了惯例中的七个Actions外，如果你需要自定群集的Action，可以这样设定：
```ruby
resources :products do
  collection do
    get  :sold
    post :on_offer
  end
  # 或
  get  :sold, :on => :collection
  post :on_offer, :on => :collection
end
```
如此便会有sold_products_path和on_offer_products_path这两个URL Helper，产生出如products/sold和products/on_offer这样的网址。

**自定特定元素路由Member**

如果需要自定对特定元素的Action：
```ruby
resources :products do
  member do
  	get :sold
  end

  # 或
  get :sold, :on => :member
end
```
如此会有sold_product_path(@product)这个URL Helper，产生出如products/123/sold这样的网址。

### 路由concern
路由concern用于声明公共路由，公共路由可以在其他资源和路由中重复使用；
如现在有一段路由格式如下：
```ruby
resources :messages do
  resources :comments
end
 
resources :articles do
  resources :comments
  resources :images, only: :index
end
```
通过定义路由concern，可以写为：
```ruby
concern :commentable do
  resources :comments
end
 
concern :image_attachable do
  resources :images, only: :index
end
```
可以在资源中使用已定义的路由concern，以避免代码重复，并在路由间共享行为：
```ruby
resources :messages, concerns: :commentable
resources :articles, concerns: [:commentable, :image_attachable]
```

以上都是如何应用的角度说的，如果对rails不太懂的人会很难理解为啥可以这样做。下面就是从理解的角度开始说：
## 深入理解
Action Pack是Rails应用的核心，包含三个Ruby模块：Action Dispatch、Action Controller和Action View。Action Dispatch负责把请求分配给控制器；Action Controller用于处理请求，得到响应；Action View供Action Controller使用，用于格式化响应。

首先我们先理一下，请求如何分派给控制器的？而在Rails中ignoble两种分派请求的方式：第一种是在需要时可以使用的详尽方式，另一种是通常使用的便利方式。

而详尽方式是指根据匹配模式、需求以及条件，直接定义url与动作之间的映射关系，而便利模式则是根据资源定义路由。两者可以搭配使用。Rails会在请求url中编码信息，通过Action Dispatch确定如何处理请求，然后将对应的控制器进行请求处理，并传入一系列其他的请求参数。

首先让我们先看一下**表现层状态转化**(Representational State Transfer，REST)：

REST架构中，服务器使用无状态的链接用于客户端通信，二者之间关于状态的所有信息都编码在请求和响应中，长期状态保存在服务器中，以一系列可识别资源的形式存在，客户端通过一系列定义良好的url访问这些资源。REST这样做的目的是为了从根本上姐耦应用机构，所以才会将资源的内容和内容的表述进行了区分。

REST架构有严格的规则，因此网络设计者知道何时以及在何处缓存请求的响应，同时REST指定的约束也会让应用易于编写，因为所有资源都通过统一的接口对外开放，应用只需要实现列出、创建、便利和删除各个资源的接口，余下的工作交给客户端就ok了。所以在REST架构中，我们只需要使用HTTP方法(GET、PUT、PATCH、POST和DELETE)去处理应用中的资源(这些资源通过url进行标示)。举个例子，比如我想获取某个资源的信息，通过url发起HTTP GET的请求，则对应REST架构中会去请求该url链接所代表的文件资源。

基于资源的路由根据上文会自动生成7个方法，而这些方法实际上对应着基本的CRUD操作。

index：返回资源列表

create：使用post请求中的数据创建一个新资源，添加到集合中

new：构建一个新资源穿个客户端，这个资源尚未保存到服务器中，可以把呢动作的作用理解为创建一个空表单，让客户端填写。

show：返回通过params[:id]标示的资源的内容

update:使用请求中的数据更新通过params[:id]标示的资源内容

edit：返回通过params[:id]标示的资源的内容，显示为表单，以便编辑

destroy： 销毁通过params[:id]标示的资源

综上Rails也用了REST约束，所以Rails不仅对应用的结构有一定的约束(MVC)，同时也对接口也有一定的约束。
