> 代码提交规范

近期发现自己项目的提交信息非常的乱，是这样子形式的：

![000000.png](https://upload-images.jianshu.io/upload_images/7154520-6e7b13eb5a061feb.png?imageMogr2/auto-orient/strip%7CimageView2/2/w/1240)

利用git提交的时候，为了方便每次commit都很随心所欲，但是在之后想看一下之前某次的commit信息的时候，就会始终找不到(内心os，这写的是啥.......)

## 提交信息的编写格式

格式如下：

```
<type>(<scope>): <subject> // subject，对commit的简短描述
// 空一行
<body>   //body对本次commit的详细描述
// 空一行
<footer> //描述一些特殊情况，不兼容变动和issue关闭
```

其中type具有以下几种形式：

```
type：提交类型，可选值如下
* work: 开发中(work in progress)
* feature：新功能(new feature)
* fix：修补bug(fix bug)
* doc：文档(documentation changes)
* style： 格式(change code format)
* refactor：重构(modify code but not feature)
* test：增加测试(test code)
* chore：构建过程或辅助工具的变动(changes don't modify src and test files, only config or tasks)
```

## commit规范

1.用一个空行隔开标题和正文

2.限制标题字数在50个字符以内

3.用大写字母写标题行

4.不要用句号结束标题行

5.在标题行使用祈使语气

例如：

- 如果被采用，这个 commit 将会重构 X 子系统以增加可读性
- 如果被采用，这个 commit 将会更新新手入门的文档
- 如果被采用，这个 commit 将移出弃用的方法
- 如果被采用，这个 commit 将发布 1.0.0 版本
- 如果被采用，这个 commit 将从 user/branch 合并 #123 pull 请求

注意下面这些非祈使格式不能工作：

- 如果被采用，这个 commit 将用 Y 修正了问题
- 如果被采用，这个 commit 将X 的改变行为
- 如果被采用，这个 commit 将对损坏事物的更多修正
- 如果被采用，这个 commit 将新酷的 API 方法

6.正文在72个字符处换行

7.**使用正文解释是什么和为什么，而不是如何做**

## git commit校验

git工具具有一个钩子，类似于web组件的生命周期，可以在执行某些操作前后触发对应的shell脚本，于git commit相关的钩子有以下4个：

- pre-commit。提交前触发，不接收参数，可以执行一些文件校验的工作。
- prepare-commit-msg。提交前触发，接收消息参数，用于在提交之前附加一些内容。
- commit-msg。提交前触发，可接收消息参数，用于检查提交消息是否符合规范。
- post-commit。提交后触发，可执行自动推送、部署代码等操作。

在这里并不打算推荐一些新的工具，如Commitizen；我们可以直接使用能和git结合的这个脚本：

```sh
#!/bin/sh

if ! grep -Eq "^(work|feature|fix|refactor|doc|test|chore|style|revert|merge):\s*\w+" "$1"; then
  exec < /dev/tty

  while true; do
    echo "Please choose a commit type(input 1~11)"
    echo "1) work(Work in progress)"
    echo "2) feature(New feature)"
    echo "3) fix(Bug Fix)"
    echo "4) refactor(A code change that neither fixes a bug nor adds a feature)"
    echo "5) doc(Documentation only changes)"
    echo "6) test(Add missing tests or correcting existing tests)"
    echo "7) chore(Changes that don\'t modify src or test files. Such as updating build tasks, package manager)"
    echo "8) style(Code Style, Changes that do not affect the meaning of the code (white-space, formatting, missing semi-colons, etc)"
    echo "9) revert(Revert to a commit)"
    echo "a) merge(Merge a branch)"
    echo "b) blog(Post a new blog article)"
    read -p "" type
    case $type in
        [1] ) prefix="work"; break;;
        [2] ) prefix="feature"; break;;
        [3] ) prefix="fix"; break;;
        [4] ) prefix="refactor"; break;;
        [5] ) prefix="doc"; break;;
        [6] ) prefix="test"; break;;
        [7] ) prefix="chore"; break;;
        [8] ) prefix="style"; break;;
        [9] ) prefix="revert"; break;;
        [a] ) prefix="merge"; break;;
	[b] ) prefix="blog";break;;
    esac
  done
  while true;do
    echo "Please input commit title"
    read -p "" title
    if [ ! -n "$title" ]; then
      continue
    fi
    echo "Please input commit description(optional)"
    read -p "" desc
    git commit -n -m "$prefix:$title" -m "$desc"
    break;
  done
fi

exit
```

将此脚本命名为`commit-msg`并放入项目的`.git/hook`中，再次提交时就会自动检测代码提交的规范了。
